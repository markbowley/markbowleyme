Markdown reference

Frontmatter fields and blog content syntax — everything you need to write and ship Astro blog posts.

Frontmatter is YAML. It lives at the very top of the file, wrapped in two sets of ---. Astro reads it at build time. Your layout accesses it via Astro.props or the frontmatter object. None of this runs in the browser — it's all build-time data.

Common fields

title
string  common
The post title. Used in <title>, OG tags, and your post list. Always wrap in quotes if it contains a colon.

Colons break YAML. Use quotes: title: "Design: a history"

--- title: "Why I ditched Figma" ---
description
string  common
The meta description. Used for SEO and social previews. Aim for 120–155 characters. You have to wire it up in your layout's <head> — Astro doesn't do it automatically.
description: "After 8 years, here's what finally replaced it."
pubDate
date  common
Publication date. Use ISO format: YYYY-MM-DD. Content collections can validate this as a real Date type if you define a schema. Without a schema, it arrives as a string.

You can also write 2024-04-12T09:00:00Z if timezone matters.

pubDate: 2024-04-12
draft
boolean  optional
Set to true to mark a post as a draft. Astro won't automatically hide it — you need to filter it out in your page query using getCollection with a filter function.

Add this to your post list query: filter: (p) => !p.data.draft

draft: true # In your .astro page: # getCollection('blog', # ({ data }) => !data.draft # )
tags
string[]  optional
Array of topic tags. YAML arrays can be written inline or as a list. No spaces in tag names unless you quote them. These are just data — you decide how to render and link them.
# Inline (compact) tags: ["design", "tools"] # Or as a list tags: - design - tools
image / heroImage
string  optional
Path to the cover image. The field name is whatever you call it — there's no Astro default. Common names: image, heroImage, cover. Use a path relative to /public or import it properly for Astro's image optimisation.
heroImage: "/images/cover.jpg"
author
string  optional
Author name. On a personal blog it's redundant — skip it unless your layout needs it for schema.org structured data or RSS.
author: "Mark Bowley"
updatedDate
date  optional
When the post was last substantially edited. Good for SEO and for showing readers the content is maintained. Same format as pubDate.
pubDate: 2024-01-10 updatedDate: 2024-04-12

YAML rules worth knowing

Quotes
Use them whenever the value contains a colon, hash, or starts with a special character. Single or double both work.
Booleans
Write as true or false, no quotes. Quoting them makes them strings.
Multiline
Use | to preserve line breaks, > to fold them into a single line.
Indentation
Always spaces, never tabs. Lists indent 2 spaces under their key.
Null / empty
Leave blank or write null. Both evaluate to undefined in your component.

A complete example

--- title: "Designing for real people" description: "Why I stopped designing in Figma and started sketching on paper." pubDate: 2025-02-20 updatedDate: 2025-02-25 heroImage: "/images/paper-sketches.jpg" tags: ["design", "process", "tools"] draft: false --- # Your markdown content starts here

Accessing in your layout

// In your BlogPost.astro layout const { title, description, pubDate, heroImage } = Astro.props; // Or via content collections (recommended) const { data } = Astro.props.entry; const { title, pubDate } = data;

Everything below the closing --- is your post content, written in standard Markdown. Astro compiles it to HTML at build time. What you write maps directly to HTML elements — your CSS styles them.

Headings

H1 – H3
Use # hashes before text. One hash = H1, two = H2, three = H3. Your post title is usually H1 — most themes render it from frontmatter, so start body copy at H2.

Don't skip heading levels (e.g. H2 → H4). Bad for screen readers and SEO.

# Post title (H1) ## Section heading (H2) ### Sub-section (H3)
renders as

Post title

Section heading

Sub-section

Emphasis

Bold & italic
Wrap text in asterisks. One pair = italic, two pairs = bold. You can combine them. Underscores also work but asterisks are more consistent across parsers.
*italic* **bold** ***bold italic***
renders as

italic    bold    bold italic

Lists

Unordered
Use - at the start of each line. Indent 2 spaces for nested items. Blank lines between items wrap each in a <p> — leave them out for tighter spacing.
- First item - Second item - Nested item - Third item
renders as
  • First item
  • Second item
    • Nested item
  • Third item
Ordered
Start each line with a number and a dot. The actual numbers don't matter — Markdown counts them for you. You can write 1. for every line and it'll render correctly.
1. First step 2. Second step 3. Third step
renders as
  1. First step
  2. Second step
  3. Third step

Links

Inline links
Square brackets for the label, round brackets for the URL. You can add a tooltip as a quoted string after the URL.

For internal Astro links, use root-relative paths like /blog/my-post — not full URLs.

[link text](https://example.com) [with tooltip](https://example.com "tip") [internal page](/blog/my-post)
renders as

link text   with tooltip   internal page

Images

Basic image
Like a link, but with a ! in front. Alt text goes in the square brackets — don't leave it empty. Path should point to your /public folder.

For automatic optimisation (WebP, lazy-loading, sizing), switch to an .mdx file and use Astro's <Image /> component instead.

![descriptive alt text] (/images/photo.jpg) # With a title attribute: ![alt text](/images/photo.jpg "caption")
renders as
[ image: descriptive alt text ]

Blockquotes

Blockquote
Prefix lines with >. Good for pull quotes or callouts. You can nest them with >> and use other markdown syntax inside.
> This is a blockquote. > It can span multiple lines. > > Even multiple paragraphs.
renders as

This is a blockquote. It can span multiple lines.


Even multiple paragraphs.

Code

Inline & fenced blocks
Single backtick for inline code. Triple backticks for a fenced block. Add the language name after the opening fence for syntax highlighting — Astro uses Shiki by default, so it works out of the box.
# Inline Use `const x = 5` in your component. # Fenced block ```js const greeting = "hello"; console.log(greeting); ```
renders as

Use const x = 5 in your component.


const greeting = "hello";
console.log(greeting);

Horizontal rule

Section divider
Three or more dashes on their own line renders as an <hr>. Leave a blank line above and below — otherwise the line above it can be read as a heading.
Some content above. --- Some content below.
renders as

Some content above.

Some content below.

Line breaks & paragraphs

Spacing rules
A blank line between blocks creates a new paragraph. A single line break in source is ignored — treated as a space. To force a line break within a paragraph, end the line with two spaces.

This trips people up constantly. If two paragraphs appear merged, you forgot the blank line between them.

# Two paragraphs First paragraph. Second paragraph. # Forced line break Line one. ← two trailing spaces Line two on a new line.
